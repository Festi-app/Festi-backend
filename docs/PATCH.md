# 사용자 (User)

- 총 관리자 계정은 축제별 1개 고정 (admin / pw 사전 생성)
- 사용자가 회원가입시 필요한 정보는 4가지
    - ID (로그인시 필요한 값)
        - 기존에 내부적으로 쓰던 UUID 타입의 id는 폐기
    - PW (Hash로 저장)
    - 전화번호 (웨이팅 연락용)
    - 이름 (부스 관리자는 대표자 이름)
- 일반 사용자 온보딩은 모바일 회원가입 화면에서만, 부스 관리자 온보딩은 회원가입 + 부스 신청을 동시에 진행
- 모든 사용자 계정은 축제에 bound됨
    - 축제마다 별도의 계정 필요 (재사용 X)

## 부스 즐겨찾기 (Favorites)

- 사용자당 주간/야간/푸드트럭 각 5개까지
- 정렬 순서를 위해 즐겨찾기 생성 시점을 기록
- 즐겨찾기 삭제는 HARD DELETE (DB에서 삭제)

# 부스 (Booth)

- 한번에 연속되는 여러 칸에 배치될 수 있음
    - BoothLocation → Booth (FK) 로 구현하고 Booth에서 역참조?
- 필드 정보
    - 이미지 (Optional)
    - 설명 (Optional)
- BoothType을 통해 부스의 운영 유형(주간/야간/푸드트럭)을 구분

## 메뉴 (Menu)

- 필드 정보
    - 이미지 (Optional)

## 푸드트럭

- 내부적으로 부스로 표현, 부스 관리자는 축제 관리자 id로 배정

## 부스 위치 (BoothLocation)

- UNIQUE ( 구역명(zone_label) + 위치 번호(index) + 운영 일자(day) )
- 프론트에서 구역별로 몇개 칸으로 나눌지 결정할 수 있음
    - BE로는 각 구역별 칸 수 정보가 전달되고, 이를 토대로 백엔드에서 BoothLocation을 생성해야 함.

# 축제 (Festival)

## 축제 일정 (FestivalDay)

- 각 일자별 주간/야간 시간대 설정

## 공지 (Notice)

- 유형 (공지/이벤트) 삭제
- 공지 상세보기 페이지 X → 공지 내용도 짧을 것으로 예상
- 상단 고정 여러개 가능
    - 여러개 고정 시 작성일 순으로 정렬

## 공연 타임라인 (Timeline)

- 축제 일자별 공연 시간표
- 필드 정보
    - 공연명
    - 아티스트 또는 팀명
    - 공연 시작 시간
    - 공연 종료 시간

# 웨이팅 (Waiting)

- 사용자당 최대 3개까지 웨이팅 등록 가능
- 웨이팅은 야간에만 가능
- 웨이팅 정보 갱신은 사용자가 직접 화면을 새로고침해서 진행
- 웨이팅 알림 (사용자 앱으로 알림)은 서버에서 전송해야 함
